# 线程的历史

一部对于 CPU 性能压榨的历史

  • 单进程人工切换：执行不同的程序需要手动切换
  • 多进程批处理：多个任务批量执行，如果一个任务阻塞，那么其他任务也无法进行下去
  • 多进程并行处理：把程序写在不同的内存位置上来回切换，即由操作系统来调度程序在 cpu 中执行
  • 多线程：一个程序内部不同任务来回切换，如 selector、epoll
  • 纤程 / 协程：绿色线程，用户管理的（而不是 OS 管理的）线程

# 进程 / 线程 / 纤程 / 协程 / 程序

计算机组成

程序：可执行文件，如 xx.exe

进程：资源分配的最小单位。一个程序可能由多个进程组成，进程启动后，操作系统会在内存中给进程分配空间

线程：程序调度执行的最小单位，多个线程共享进程中的资源

  • 单线程：只有一条执行路径，一个执行任务
  • 多线程：有多条执行路径在同时运行，多个执行任务

线程的切换（context switch）：暂停执行的线程存放于缓存中，通过操作系统来进行调度，将线程重新加载到 CPU 去执行，线程切换消耗 CPU 资源。关于线程切换有如下几个问题：

1、单核 CPU 设定多线程是否有意义？

当一个线程在执行期间处于等待状态时，其他线程可以获得 CPU 资源继续执行，充分利用 CPU 资源。所以单核 CPU 设定多线程是有意义的。

2、工作线程数是不是设置的越大越好？

线程的切换需要消耗 CPU 资源，所以在能完成任务的情况下，线程数越少越好

3、工作线程数（线程池中线程数量）设置多少合适？

可以根据 CPU 核数来设定线程数，同时，为了安全起见，要留出几个核来作为应急情况其他程序的执行。

关于 W/C 可以使用性能分析工具来得出

并发：一个时间段内，多个进程通过竞争 CPU 资源交替执行，微观上进程不是同时执行的，宏观上进程是同时执行的

并行：同一个时刻，多个进程在多个处理器上同时执行，微观、宏观上进程都是同时执行的

# 线程的 6 种状态

1、New：线程刚刚创建，还没有启动

2、RUNNABLE：可运行状态（调用 start () 方法后进入可运行状态），由线程调度器可以安排执行。包括 READY 和 RUNNING 两种细分状态，READY 状态的线程主动调用 Thread.yiled () 方法让出 CPU 资源，等待线程调度器重新调起线程后获得 CPU 时间片继续执行，RUNNING 状态的线程是正在执行的线程，正在执行的线程的执行时间就是 CPU 时间片的大小

3、WAITING：线程等待被唤醒，调用 Object 的 notify () 或者 notifyAll () 方法后可以唤醒线程

4、TIMED WAITING：线程等待指定时间后自动唤醒

5、BLOCKED：线程被阻塞，正在等待 synchronied 锁，获得锁后进入 RUNNABLE 状态

6、TERMINATED：线程结束，run () 方法执行完成或者异常退出

# wait 和 sleep 的区别

  1. 所属类不同：wait 是 Object 类的方法，sleep 是 Thread 的静态方法
  2. 使用范围不同：wait 只能在同步控制方法或者同步控制块里面使用，而 sleep 可以在任何地方使用
  3. 是否释放锁：在调用 sleep () 方法的过程中，线程不会释放对象锁；而当调用 wait () 方法的时候，线程会放弃对象锁，进入等待此对象的等待锁池，只有针对此对象调用 notify () 方法后本线程才进入对象锁池，准备再次获得锁
  4. sleep，wait 调用后都会暂停当前线程并让出 cpu 的执行时间，但不同的是 sleep 不会释放当前持有的对象的锁资源，到时间后会继续执行，而 wait 会放弃所有锁并需要 notify/notifyAll 后重新获取到对象锁资源后才能继续执行。
  5. 其次，当对 sleep ()、wait ()、join () 设置 interrupt 时将抛出异常 InterruptedException，且状态位被重置
  6. sleep 必须捕获异常，wait、notify、notifyAll 不需要捕获异常

例如：

Thread.Sleep (1000) 意思是在未来的 1000 毫秒内本线程不参与 CPU 竞争，1000 毫秒过去之后，这时候也许另外一个线程正在使用 CPU，那么这时候操作系统是不会重新分配 CPU 的，直到那个线程挂起或结束，即使这个时候恰巧轮到操作系统进行 CPU 分配，那么当前线程也不一定就是总优先级最高的那个，CPU 还是可能被其他线程抢占去。另外值得一提的是 Thread.Sleep (0) 的作用，就是触发操作系统立刻重新进行一次 CPU 竞争，竞争的结果也许是当前线程仍然获得 CPU 控制权，也许会换成别的线程获得 CPU 控制权。

wait (1000) 表示将锁释放 1000 毫秒，到时间后如果锁没有被其他线程占用，则再次得到锁，然后 wait 方法结束，执行后面的代码，如果锁被其他线程占用，则等待其他线程释放锁。注意，设置了超时时间的 wait 方法一旦过了超时时间，并不需要其他线程执行 notify 也能自动解除阻塞，但是如果没设置超时时间的 wait 方法必须等待其他线程执行 notify。

# 阻塞的三种情况

等待阻塞（o.wait 等待对列）

运行 (running) 的线程执行 o.wait () 方法，JVM 会把该线程放入等待队列 (waitting queue) 中。

同步阻塞 (lock）

运行 (running) 的线程在获取对象的同步锁时，若该同步锁被别的线程占用，则 JVM 会把该线程放入锁池 (lock pool) 中。

其他阻塞 (sleep/join)

运行 (running) 的线程执行 Thread.sleep (long ms) 或 t.join () 方法，或者发出了 I/O 请求时，JVM 会把该线程置为阻塞状态。当 sleep () 状态超时、join () 等待线程终止或者超时、或者 I/O 处理完毕时，线程重新转入可运行 (runnable) 状态。

# 终止线程的四种方式

  1. 正常运行结束

run () 或 call () 方法执行完成，线程正常结束。

  1. 使用退出标志退出线程

一般 run () 方法执行完，线程就会正常结束，然而，常常有些线程是伺服线程。它们需要长时间的

运行，只有在外部某些条件满足的情况下，才能关闭这些线程。使用一个变量来控制循环，例如：

最直接的方法就是设一个 boolean 类型的标志，并通过设置这个标志为 true 或 false 来控制 while 循环是否退出，代码示例：

public class ThreadSafe extends Thread {
public volatile boolean exit = false;
public void run() { while (!exit){
//do something
}
}
}

定义了一个退出标志 exit，当 exit 为 true 时，while 循环退出，exit 的默认值为 false. 在定义 exit 时，使用了一个 Java 关键字 volatile，这个关键字的目的是使 exit 同步，也就是说在同一时刻只能由一个线程来修改 exit 的值。

  1. Interrupt 方法结束线程

使用 interrupt () 方法来中断线程有两种情况：

1 线程处于阻塞状态：如使用了 sleep, 同步锁的 wait,socket 中的 receive,accept 等方法时，会使线程处于阻塞状态。当调用线程的 interrupt () 方法时，会抛出 InterruptException 异常。阻塞中的那个方法抛出这个异常，通过代码捕获该异常，然后 break 跳出循环状态，从而让我们有机会结束这个线程的执行。通常很多人认为只要调用 interrupt 方法线程就会结束，实际上是错的， 一定要先捕获 InterruptedException 异常之后通过 break 来跳出循环，才能正常结束 run 方法。

2 线程未处于阻塞状态：使用 isInterrupted () 判断线程的中断标志来退出循环。当使用 interrupt () 方法时，中断标志就会置 true，和使用自定义的标志来控制循环是一样的道理。

public class ThreadSafe extends Thread {
public void run() {
while (!isInterrupted()){ //非阻塞过程中通过判断中断标志来退出
try{
Thread.sleep(51000);//阻塞过程捕获中断异常来退出
}catch(InterruptedException e){
e.printStackTrace();
break;//捕获到异常之后，执行 break 跳出循环
}
}
}
}
  1. stop 方法终止线程（线程不安全）

程序中可以直接使用 thread.stop () 来强行终止线程，但是 stop 方法是很危险的，就象突然关闭计算机电源，而不是按正常程序关机一样，可能会产生不可预料的结果，不安全主要是：thread.stop () 调用之后，创建子线程的线程就会抛出 ThreadDeatherror 的错误，并且会释放子线程所持有的所有锁。一般任何进行加锁的代码块，都是为了保护数据的一致性，如果在调用 thread.stop () 后导致了该线程所持有的所有锁的突然释放 (不可控制)，那么被保护数据就有可能呈现不一致性，其他线程在使用这些被破坏的数据时，有可能导致一些很奇怪的应用程序错误。因此，并不推荐使用 stop 方法来终止线程。

# synchronized

可以锁 代码块、方法、对象。注意不能锁 String 常量以及基本数据类型（如 Integer、Long 等），因为这些类型的值变化后就是一个新的对象。

锁升级：

偏向锁：当只有一个线程使用 synchronized 给对象加锁时，其实在对象头中只是记录这个线程的 id，并没有实际加锁，此时称为偏向锁（即无锁，对象头中锁标志位为 01），效率较高；

自旋锁：当有线程争用锁时，锁升级为自旋锁（默认自旋 10 次，自旋的意义是让线程自旋等待去获取锁而不是阻塞，减少因线程阻塞带来的线程切换（线程挂起、恢复）），线程自旋将占用 CPU 时间，自旋结束后进入等待队列阻塞，所以，自旋锁适合少数线程争用锁的情况，否则会占用 CPU 资源过高；还有一个自适应自旋锁的概念，自适应自旋假定不同线程持有同一个锁对象的时间基本相当，竞争程度趋于稳定，因此，可以根据上一次自旋的时间与结果调整下一次自旋的时间；

轻量级锁：当线程自旋指定次数后仍无法获得锁时，将升级为轻量级锁（对象头中锁标志位为 00），轻量级锁的目标是，减少无实际竞争情况下，使用重量级锁产生的性能消耗，包括系统调用引起的内核态与用户态切换、线程阻塞造成的线程切换等；

重量级锁：当锁竞争激励时，将升级为重量级锁（对象头中锁标志位为 10）去操作系统请求资源。

# CAS（无锁优化，自旋）

Compare And Swap，CAS 是 CPU 指令级别的支持，在保证原子的过程中不能被打断

cas (期望值，更新值)，例如：cas (0,1)，如果当前值为 0，那么证明没有被其他线程修改过，所以可以拿来直接计算，如加 1，下一次加 1 计算就变成了 cas (1,2)

CAS 的问题

ABA 问题：一个线程在使用变量的过程中，变量被另外一个线程更改后又改回了原始值。（该问题不会影响到基本类型，对引用类型会有影响）

解决：给变量加版本号

A 1.0 (变量版本号起始为 1.0)

B 2.0（被修改为 B 后版本号变成了 2.0）

A 3.0（再次被修改回去变为 A 后，版本号变为了 3.0）

# Atomic、LongAdder、Synchronized

测试这三种方式下线程执行的速度快慢：

Synchronized 是加锁的，会有线程的竞争，势必就会造成异步线程的阻塞，而阻塞会带来线程的切换（线程挂起、切换）上的开销；Atomic 是用 CAS 来实现的，线程通过自旋等待去尝试获取锁而不是阻塞，这就解决了线程切换的开销，所以 Atomic 相对 Synchronized 来说，效率比较高，但是当线程数比较多，锁竞争激烈的情况下，过多线程的自旋会占用过多 CPU 时间，导致 CAS 性能下降，所以，CAS 适合线程数比较少的情况；LongAdder 内部使用了分段锁的概念，即将所有线程分为几个部分，每个部分中的线程被分别加锁，将各个部分中的线程的计算结果加起来得到最后结果。

# Synchronized 与 ReentrantLock 的区别

  • synchronized 自动加锁、自动释放锁；reentrantLock 手动加锁、手动释放锁

  • synchronized 有锁升级的过程；reentrantLock 是 CAS 的实现，有等待队列 AQS 的概念

# ReentrantLock 源码

ReentrantLock 的内部类有 Sync、NonfairSync、FairSync，后两者都继承了 Sync，而 Sync 继承了 AQS，即 AbstractQueedSynchronizer。AQS 维护了一个双向链表，用于存放请求锁的线程，每个线程调用 AQS 中的 acquire () 方法（具体实现在子类 NonfairSync、FairSync 中）来请求获得锁，如果当前的 state=0（使用了 volatile 修饰），那么请求的线程可以获得锁，如果 state=1，那么线程进入等待队列（链表）

# 强引用

new 一个对象时，这种将引用赋值给对象的情形就是强引用（即正常的引用）

# 软引用

当 new 一个对象后，如果该对象还指向其他引用，那么这个引用就是软引用，只有当内存不足时才会回收该软引用。软引用非常适合缓存使用。

# 弱引用

如果有强引用指向弱引用，那么强引用被回收后，弱引用也会被回收，一般用于容器中。ThreadLocal 也是弱引用的应用之一，ThreadLocal 中维护的 map 中的 key 是弱引用（WeakHashMap），为了能够让 ThreadLocal 对象及时回收，防止内存泄漏

# 虚引用

管理堆外内存（直接内存 DirectByteBuffer，Netty 中有引用），虚引用遇到垃圾回收时会立即被回收，回收的虚引用被放在一个队列中，通过队列来清理该内存（因为堆外内存，jvm 无法回收）。在 java 中，可以通过 Unsafe 类来直接分配内存（allocateMemory ()）以及回收内存（freeMemory ()）

方式一：使用 volatile 实现（本质上无法实现，加了 sleep 后可以实现）
