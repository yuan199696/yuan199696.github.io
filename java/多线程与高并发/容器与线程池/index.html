# 从 HashTable 到 ConcurrentHashMap

HashTable->HashMap->Collections.synchronizedMap(new HashMap<UUID, UUID>())->ConcurrentHashMap

HashTable 的每一个方法都加了锁（synchronized），导致读取效率较低

使用 synchronizedMap () 可以对一个 HashMap 进行加锁，是对对象加锁，所以锁的粒度比 HashTable 中的方法锁的粒度要小，所以读取的效率会有所提高

ConcurrentHashMap 的优势在于读取非常快，而插入操作要比 HashTable、SynchronizedHashMap 的效率慢

# 从 Vector 到 Queue

Vector 中的很多方法加了锁（synchronized），所以效率不是很高，Vector 和 HashTable 是最初的集合，由于其性能问题，逐渐被淘汰

如果在多线程中，对于操作单个元素，那么可以优先考虑 Queue，而不是 List

# 线程池

线程池可以分为两大类：ThreadPoolExecutor（执行一般任务）、ForkJoinPool（执行可拆分的任务）

# Executor

执行器，只有一个方法 execute，执行线程任务，留给子类去实现

# ExecutorService

继承了 Executor，定义了一些操作线程池的方法。其子类主要是各种线程池的实现。通过 submit () 可以将任务（如 Callable、Runnable 等线程任务）提交到线程池，并返回一个存放任务执行结果的 Future 对象，进一步通过 Future 对象的 get () 可以获取执行结果。其中，submit 是异步的，即不会阻塞等待任务的返回结果。

# Callable

一个可以执行的任务，相比 Runnable 来说，Callable 有返回结果

# Future

用于存放异步任务（将来产生的结果）的计算结果，可通过 get () 获取结果

# FutureTask

RunnableFuture 的一个实现类，既是一个封装任务的 Future，又是一个任务（线程任务，Runnable），即任务执行后，执行结果存在于 FutureTask 自己的对象中

# CompletableFuture

可以组合多个任务的返回结果

# ThreadPoolExecutor

通过 ThreadPoolExecutor 可以创建一个线程池，该类继承自 AbstractExecutorService，而 AbstractExecutorService 实现了 ExecutorService 接口。一个线程池中有线程集合和任务集合（队列）。

JUC 包中提供了一个工具类 Executors，通过该类可以调用一些封装好的线程池方法来创建线程池

# 定义任务

任务一般实现了 Runnable 接口或 Callable 接口

# 自定义线程池

参数一：核心线程数

参数二：最大线程数（核心线程 + 非核心线程数），当核心线程不够用时，就扩展一些线程进入线程池执行任务

参数三：非核心线程过期时间，如果在这个时间内非核心线程没被利用，那么就从线程池回收该线程

参数四：非核心线程允许的空闲时间的单位

参数五：任务队列（任务一般实现了 Runnable 接口或 Callable 接口），指定允许的任务个数。不同的队列会形成不同的线程池

参数六：产生线程的工厂，可以自定义，需要实现 ThreadFactory 接口。

参数七：拒绝策略（JDK 默认提供了四种），可以自定义。当线程池所有线程都处于忙碌状态，且任务队列已经满了的情况下，这时候仍然有任务请求执行时，就需要拒绝策略以一定的方式拒绝任务。

  • AbortPolicy：抛异常：RejectedExecutionException
  • DiscardPolicy：扔掉新来的任务，不抛异常
  • DiscardOldestPolicy：扔掉排队时间最久的任务，默认的拒绝策略
  • CallerRunsPolicy：调用者处理任务（哪个线程调用线程池的 execute ()，那么该线程去执行任务）

# SingleThreadPool

线程池中只有一个线程，可以保证任务队列中任务的执行顺序。单线程的线程池比单独创建一个线程的好处在于：

  • 有任务队列
  • 线程池可以自动管理线程的生命周期

# CacheThreadPool

可以看到该线程池中的任务队列使用了 SynchronousQueue，该队列大小为 0，即一旦该队列中有任务，那么就立即交给线程去执行该任务。该线程池适合任务任务数量有起伏、不确定的情况。

# FixedThreadPool

适合任务数量比较平稳的情况下使用。可以让任务达到并行执行

# ScheduledThreadPool

# ForkJoinPool

  • 用很少的线程可以执行很多的任务（子任务）

# 线程池大小

W/C：即 wait/computation 的比值，如果是 IO 密集型，那么传输的耗时就导致等待时间较长，此时需要更多的线程去并发工作；如果是 CPU 密集型，那么等待时间较短，计算比较快，此时需要的线程数相对于 IO 密集型来说较少。而实际线程数需要压测得出。

# 特点

  • Disruptor 是数组（环形数组）实现的

  • 无锁，高并发，使用环形 Buffer，直接覆盖（不用清除）旧的数据，降低 GC 频率

  • 实现了基于事件的生产者消费者模式（观察者模式）

# RingBuffer

  • 环形数组

  • RingBuffer 的序号（sequence），指向下一个可用的元素，只需要维护该序号即可

  • 采用数组实现（数组查找比链表快），没有首尾指针

    假如长度为 8，当添加到第 12 个元素的时候在哪个序号上呢？用 12%8 决定（第四个位置，环形数组下标为 3）。当 Buffer 被填满的时候到底是覆盖还是等待，由 Producer（生产者的策略）决定。长度设为 2 的 n 次幂，利于二进制计算，例如：12%8 = 12 & (8 - 1) ，即 pos = num & (size -1)

# Disruptor 开发步骤

  1. 定义 Event，环形队列中存放的是一个个的 Event，即需要处理的元素

  2. 定义 Event 工厂，用于填充队列。Disruptor 初始化时，会调用 Event 工厂，对 RingBuffer 进行内存的提前分配。

  3. 定义 EventHandler，处理容器中的元素，相当于消费者（可以有多个消费者，分别处于多个线程）

# ProducerType

生产者线程模式，ProducerType 有两种模式 Producer.MULTI 和 Producer.SINGLE，默认是 MULTI，表示在多线程模式下产生 sequence，如果确认是单线程生产者，那么可以指定 SINGLE，效率会提升，因为单线程不需要加锁。如果是多个生产者（多线程），但模式指定为 SINGLE，那么生产出来的元素可能被其他线程生产出来的元素覆盖

# 等待策略

RingBuffer 满了后生产者线程（producer）执行的策略

  1. (常用）BlockingWaitStrategy：通过线程阻塞的方式，等待生产者唤醒，被唤醒后，再循环检查依赖的 sequence 是否已经消费。

  2. BusySpinWaitStrategy：线程一直自旋等待，可能比较耗 cpu

  3. LiteBlockingWaitStrategy：线程阻塞等待生产者唤醒，与 BlockingWaitStrategy 相比，区别在 signalNeeded.getAndSet, 如果两个线程同时访问一个访问 waitfor, 一个访问 signalAll 时，可以减少 lock 加锁次数.

  4. LiteTimeoutBlockingWaitStrategy：与 LiteBlockingWaitStrategy 相比，设置了阻塞时间，超过时间后抛异常。

  5. PhasedBackoffWaitStrategy：根据时间参数和传入的等待策略来决定使用哪种等待策略

  6. TimeoutBlockingWaitStrategy：相对于 BlockingWaitStrategy 来说，设置了等待时间，超过后抛异常

  7. （常用）YieldingWaitStrategy：尝试 100 次，然后 Thread.yield () 让出 cpu

  8. （常用）SleepingWaitStrategy : sleep

# 并行与并发

并行（parell）：多个任务在多个 CPU 上同时处理执行，即并行是指任务执行

并发（concurrent）：多个任务在一个 CPU 上切换执行，即并发是指任务提交（不一定执行）
