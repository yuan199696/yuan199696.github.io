# Session/Cookies/Token

浏览器请求访问服务端时，服务端会为浏览器的请求生成一个 session id，用来标识浏览器与服务端之间的会话，生成的 session id 同时会存放在浏览器上一份，称之为 cookies。如果不需要服务端生成的 session id，那么可以手动生成一个唯一标识来保证会话的唯一性，这个唯一标识就是 Token。

# 跨域的解决

浏览器会对不同域名或不同端口的 URL 之间的请求形成一个安全策略，即同源策略，即一个会话必须发生在相同协议、相同域名（一级域名）、相同端口的 URL 请求下才能顺利完成请求，否则不同的 URL 之间无法完成任何会话中的请求（如 js 的异步请求等）。但是，对于现在前后端分离的项目，前端与后端用于发送请求的 URL 必定是不同源的（跨域名的、跨端口的），所以需要突破浏览器的安全策略使得不同源的 URL 之间的请求能够在同一个会话下进行。常用的跨域解决方案是 JSONP、CORS。

  • JSONP：此方法只能发起 GET 请求，通过 jsonp 发送的请求，会随带 cookie 一起发送。
  • CORS：在浏览器中指定 Origin 来源，如果在服务器接受范围，请求则成功。CORS 与 JSONP 的使用目的相同，但是比 JSONP 更强大。

JSONP 只支持 GET 请求，CORS 支持所有类型的 HTTP 请求。JSONP 的优势在于支持老式浏览器，以及可以向不支持 CORS 的网站请求数据。

# XSS 与 CSRF

# XSS

XSS 攻击是指在 html 中嵌入能发起外站请求的脚本。主要应用于流量攻击。解决办法：可以将内容进行转义后显示。

# CSRF

伪装请求到另一个站点，例如：通过 img 标签的 src 请求到另一个站点。这种攻击都是 get 请求，所以可以将所有请求替换为 post 请求来避免 CSRF 攻击。

解决方案：敏感操作加验证码；每次请求带一个 token。

# 密码安全性

对于 MD5 算法来说，现在是不安全的，因为可以通过彩虹表分段来碰撞解密。所以，一般的做法是给密码加盐后再通过算法去生成密码，关键就在于盐的选择，固定盐肯定是不安全的，可以用随机数作为盐，即 hash (明文 + 随机数)，但这样做的问题是每次相同的明文由于随机数的不同导致加密结果不同，所以造成了无法匹配的问题。解决办法是将每次生成的随机数记录下来加在加密结果的后边或前面或中间后存入数据库，将每次请求的明文密码加上数据库中已有密码后面的随机数再次执行相同的算法 hash (明文 + 随机数) 后的到结果去匹配数据库中的结果。在此结果上还可以进行多次加密，即 hash (hash (明文 + 随机盐) + 随机盐)，如 SHA-1、SHA-256 等，这样的加密方式是非常安全的，以目前的计算机算力想要解密是不可能的，除非量子计算机。
